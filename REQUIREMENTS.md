# Student ID Card Issuing System - Requirements

## 1. Introduction

This document defines the functional and non-functional requirements of the Student ID Card Issuing System.

The system provides a secure method for issuing digital student ID cards. Students must verify their identity using a one-time OTP generated by the registrar before uploading their photo and generating their ID card.

---

# 2. Users and Roles

## 2.1 Registrar (Administrator)

The registrar manages student information and controls the ID issuing process.

Responsibilities:

* Manage student records
* Generate OTP verification codes
* Monitor student verification status
* Manage system information

---

## 2.2 Student

The student uses the system to obtain their ID card.

Responsibilities:

* Verify identity using Student ID and OTP
* Upload personal photo
* Preview ID card
* Download generated PDF ID card

---

# 3. Functional Requirements

## 3.1 Authentication Requirements

The system shall:

* Allow registrar login
* Validate administrator credentials
* Protect administrative functions from unauthorized users
* Maintain user sessions securely

---

# 3.2 Student Management Requirements

The system shall allow the registrar to:

* Create student records
* View student information
* Update student information
* Delete student records
* Search students

Student information includes:

* Student ID
* First name
* Last name
* Department
* Program
* Academic year
* Email
* Phone number

---

# 3.3 OTP Management Requirements

The system shall:

* Generate unique OTP codes
* Associate OTP codes with specific students
* Store OTP creation time
* Set OTP expiration time
* Allow OTP verification
* Prevent reuse of used OTP codes

OTP rules:

* OTP length: 6 digits
* OTP can only be used once
* Expired OTP cannot be accepted

---

# 3.4 Student Verification Requirements

The system shall allow students to:

* Enter Student ID
* Enter OTP code
* Submit verification request

The system shall:

* Validate student existence
* Validate OTP correctness
* Check OTP expiration
* Check OTP usage status

---

# 3.5 Photo Upload Requirements

The system shall allow verified students to:

* Upload their personal photo
* Replace photo before final submission

The system shall validate:

* File type
* File size
* File extension

Supported formats:

* JPG
* JPEG
* PNG

---

# 3.6 ID Card Generation Requirements

The system shall:

* Generate a digital student ID card
* Include student information
* Include student photo
* Generate PDF format
* Allow students to download the PDF

The generated ID card should contain:

* University name
* Student photo
* Student name
* Student ID
* Department
* Program
* Academic year

---

# 3.7 Data Management Requirements

The system shall:

* Store student information
* Store OTP information
* Store administrator information
* Maintain data consistency

---

# 4. Non-Functional Requirements

## 4.1 Security

The system should:

* Protect student information
* Hash passwords securely
* Validate user input
* Prevent unauthorized access
* Secure API endpoints
* Validate uploaded files

---

## 4.2 Performance

The system should:

* Respond quickly to user requests
* Support multiple students accessing the system
* Generate PDF files efficiently

---

## 4.3 Reliability

The system should:

* Maintain accurate student records
* Prevent duplicate ID generation
* Handle errors gracefully

---

## 4.4 Usability

The system should:

* Provide a simple interface
* Work on desktop and mobile browsers
* Provide clear error messages

---

## 4.5 Maintainability

The system should:

* Use clean architecture
* Separate frontend and backend
* Follow coding standards
* Include documentation

---

# 5. Business Rules

## Student ID Rules

* Each student must have a unique student ID.
* Only registered students can receive an ID card.

---

## OTP Rules

* OTP is generated only by the registrar.
* OTP belongs to one student only.
* OTP cannot be reused after successful verification.
* OTP expires after a defined period.

---

## Photo Rules

* Students are responsible for uploading their own photos.
* Photos must meet system requirements.
* Invalid files must be rejected.

---

# 6. Constraints

The system requires:

* Internet connection
* Modern web browser
* Backend server
* MariaDB database

---

* Digital signatures
* Email OTP delivery
* Mobile application
